# פרומפט איפיון: דאשבורד ניהול עסקאות — Safe Capital

## סקירה כללית

בנה דאשבורד ניהול עסקאות נדל"ן עבור אתר Safe Capital.
הדאשבורד כולל שלושה ממשקים:
1. **דאשבורד אדמין ראשי** (`/admin`) — Super Admin יחיד. שליטה מלאה על כל המערכת, כולל ניהול הרשאות משתמשים.
2. **דאשבורד מנהלים/שותפים** (`/admin`) — שותפים עם הרשאות ניהול עסקאות (ללא שינוי הרשאות). אותו ממשק כמו האדמין, אך ללא גישה לניהול משתמשים.
3. **דאשבורד משקיעים** (`/investor`) — [שלב 2] ממשק read-only למשקיעים שבו הם רואים פרטים על הנכסים שהשקיעו בהם, התקדמות פיננסית, תמונות וסרטונים מהשטח.

הדאשבורד נפרד לחלוטין מהאתר הציבורי — נגיש רק עם הרשאות מתאימות.

### היררכיית הרשאות (Roles)

| Role | תיאור | הרשאות |
|------|--------|--------|
| **super_admin** | בעל האתר (יחיד) | הכל + ניהול משתמשים + שינוי הרשאות |
| **manager** | שותפים/מנהלים | ניהול עסקאות, תזרים, מסמכים, משקיעים. ללא ניהול הרשאות |
| **investor** | משקיעים [שלב 2] | צפייה בלבד בעסקאות שמשויכות אליהם |

---

## טכנולוגיה

- **Frontend**: אותו stack כמו האתר הקיים — HTML + Tailwind CSS + Vanilla JS
- **Backend**: Node.js + Express (או JSON file-based לשלב ראשון)
- **בסיס נתונים**: SQLite (קובץ יחיד, פשוט, ללא תלות בשרת DB חיצוני) — או JSON files לשלב MVP
- **אימות**: מערכת משתמשים עם 3 roles — super_admin, manager, investor. JWT session. סיסמאות נשמרות כ-hash (bcrypt).
- **חילוץ נתונים אוטומטי**: AI/OCR לחילוץ נתונים ממסמכים שהמשתמש מעלה (spreadsheets, PDFs, תמונות) — המערכת מזינה את הנתונים לבד
- **עיצוב**: RTL, עברית, מיתוג Safe Capital (אותם צבעים, פונטים, ורוח עיצובית)

## מבנה הדאשבורד

### 1. מסך התחברות

- שדה אימייל + סיסמה
- לוגו Safe Capital
- עיצוב מינימליסטי ומיתוגי
- המערכת מזהה את ה-role ומפנה לדאשבורד המתאים:
  - super_admin / manager → `/admin/`
  - investor → `/investor/` [שלב 2]

### 1.5. מסך ניהול משתמשים (super_admin בלבד) — `/admin/users`

- רשימת כל המשתמשים במערכת
- כפתור "משתמש חדש +"
- לכל משתמש: שם, אימייל, role, סטטוס
- **רק ה-super_admin יכול**:
  - ליצור משתמשים חדשים (מנהלים או משקיעים)
  - לשנות role של משתמש
  - להשעות/להפעיל משתמש
  - לאפס סיסמה

### 2. מסך ראשי — רשימת עסקאות

- טבלה/רשימת כרטיסים של כל העסקאות
- לכל עסקה: שם הנכס, כתובת, סטטוס נכס, סטטוס גיוס, סכום כולל
- כפתור "עסקה חדשה +"
- לחיצה על עסקה → נכנסים למסך עריכת העסקה

### 2.5. תהליך יצירת עסקה חדשה — העלאת מסמכים וחילוץ אוטומטי

במקום מילוי ידני של עשרות שדות, המערכת מנחה את המשתמש להעלות מסמכים והיא מחלצת את הנתונים לבד.

#### שלב 1: העלאת מסמכים (Wizard)
המערכת מציגה רשימת מסמכים נדרשים ומבקשת מהמשתמש להעלות אותם:

| מסמך נדרש | סוג קובץ צפוי | מה המערכת מחלצת ממנו |
|------------|----------------|----------------------|
| **מחשבון/תכנית עסקית** | XLSX / CSV | כל הנתונים הפיננסיים: מחיר רכישה, עלויות שיפוץ (כל קטגוריה + line items), ARV, עלויות מימון, עלויות החזקה, מחיר מכירה צפוי, רווח, ROI, מפרט לפני/אחרי (חדרים, שטח), משך פרויקט |
| **חוזה קבלן** | PDF | שם קבלן, סכום חוזה, לוח תשלומים (draw schedule), היקף עבודות |
| **הסכם הלוואה / Loan Application** | PDF | סכום הלוואה, ריבית, תנאי מימון, שם המלווה |
| **מסמכי סגירה (Closing Documents)** | PDF (סרוק) | מחיר רכישה סופי, תאריך סגירה, עלויות סגירה, מספר חלקה (Parcel ID) |
| **צילומי Comps (עסקאות דומות)** | PNG / JPG / PDF | כתובת, מחיר מכירה, ימים בשוק (OCR מתמונות מסך) |
| **תמונות לפני שיפוץ** | JPG / PNG / WebP | תמונות הנכס לפני — נשמרות כגלריה |
| **הדמיות אדריכליות** | JPG / PNG / PDF | הדמיות אחרי — נשמרות כגלריה |
| **תמונות אחרי שיפוץ** | JPG / PNG / WebP | תמונות הנכס אחרי — נשמרות כגלריה |

#### שלב 2: חילוץ אוטומטי
- המערכת קוראת את הקבצים ומחלצת את הנתונים:
  - **XLSX**: קריאת תאים ישירה — מיפוי שדות מוגדר מראש (לפי מבנה המחשבון הקיים)
  - **PDF טקסטואלי**: חילוץ טקסט + AI parsing לזיהוי שדות
  - **PDF סרוק / תמונות**: OCR → טקסט → AI parsing
- המערכת ממלאת את כל השדות בטאבים אוטומטית

#### שלב 3: אישור ועריכה
- המערכת מציגה את הנתונים שחולצו **לפני שמירה**
- שדות שהמערכת לא בטוחה לגביהם מסומנים בצהוב (confidence נמוך)
- המשתמש מאשר, מתקן אם צריך, ושומר
- לאחר האישור — הנתונים נשמרים בטאבים הרלוונטיים

#### שלב 4: העלאות נוספות
- בכל שלב אפשר להעלות מסמכים נוספים והמערכת תעדכן את הנתונים
- מסמכים שכבר הועלו מסומנים ב-✓
- אפשר תמיד לעבור לעריכה ידנית בטאבים

> **חשוב**: העלאת מסמכים היא הדרך המועדפת להזנת נתונים, אבל כל שדה ניתן לעריכה ידנית אחר כך בטאבים.

#### טאב ב׳: מחשבון פיננסי — תכנון מול ביצוע

טאב זה מנהל את הנתונים הפיננסיים בשני מצבים: **תכנון** (planned) ו**בפועל** (actual), עם השוואה אוטומטית ביניהם.

##### נתונים ראשיים
| שדה | תכנון | בפועל | הערות |
|------|--------|--------|-------|
| מחיר רכישה | מספר ($) | מספר ($) | בפועל מתעדכן מתזרים או ידנית |
| שווי עתידי (ARV) | מספר ($) | מספר ($) | בפועל = מחיר מכירה אמיתי (כשנמכר) |
| מחיר מכירה צפוי | מספר ($) | מספר ($) | + טולטיפ הסבר (טקסט חופשי) |

> **שמירת snapshot**: ברגע יצירת העסקה (או העלאת המחשבון), כל הנתונים הפיננסיים נשמרים כ-"תכנון" (snapshot). ערכי התכנון **לא משתנים** לאחר מכן אלא אם המשתמש מעדכן אותם במפורש.

##### מבנה עלויות — קטגוריות ותת-קטגוריות

המבנה הוא דו-רמתי: **קטגוריה ← פריטים (line items)**. לכל פריט יש עמודת "תכנון" ועמודת "בפועל".

**קטגוריות ברירת מחדל** (נוצרות אוטומטית מהמחשבון, ניתנות לעריכה):

| קטגוריה | דוגמאות לפריטים |
|----------|-----------------|
| **עלויות רכישה** | מחיר רכישת הנכס, עלויות סגירה, בדיקות ופיקוח |
| **עלויות שיפוץ** | הריסה ובנייה, חומרי גמר, פיתוח חוץ |
| **עלויות החזקה ותכנון** | ארנונה וביטוח, תכנון אדריכלי, תשתיות |
| **עלויות מימון** | ריבית הלוואה, עמלות פתיחה |
| **עלויות מכירה** | עמלות מתווכים, עלויות סגירת מכירה |

- המנהל יכול **ליצור קטגוריות חדשות** או **למחוק/לערוך** קיימות
- המנהל יכול **להוסיף/למחוק פריטים** בכל קטגוריה
- סכום הקטגוריה מחושב אוטומטית (סכום הפריטים)

##### לכל פריט עלות:
| שדה | סוג | הערות |
|------|------|-------|
| שם פריט | טקסט | לדוגמה: "הריסה ובנייה" |
| סכום מתוכנן | מספר ($) | מוזן ביצירה, לא משתנה |
| סכום בפועל | מספר ($) | מתעדכן מתזרים מזומנים או ידנית |
| חריגה | מחושב | בפועל − תכנון (ירוק אם שלילי/0, אדום אם חיובי) |
| % חריגה | מחושב | (חריגה / תכנון) × 100 |

##### דאשבורד השוואה תכנון מול ביצוע

תצוגת סיכום ויזואלית בראש הטאב:

| מדד | תכנון | בפועל | חריגה | סטטוס |
|------|--------|--------|--------|--------|
| סך השקעה כולל | $X | $Y | $Z | ירוק/צהוב/אדום |
| עלויות רכישה | $X | $Y | $Z | ירוק/צהוב/אדום |
| עלויות שיפוץ | $X | $Y | $Z | ירוק/צהוב/אדום |
| עלויות מימון | $X | $Y | $Z | ירוק/צהוב/אדום |
| ... (כל קטגוריה) | | | | |
| רווח נקי צפוי | $X | $Y | $Z | |

**כללי צביעת סטטוס:**
- **ירוק**: בפועל ≤ תכנון (עומדים בתקציב או מתחת)
- **צהוב**: חריגה של עד 10% מהתכנון
- **אדום**: חריגה מעל 10% מהתכנון

##### חיבור תזרים → עלויות בפועל

כאשר מוזנת תנועת תזרים מזומנים (טאב ד׳) עם קטגוריה ותת-קטגוריה, המערכת **מעדכנת אוטומטית** את הסכום "בפועל" בפריט העלות המתאים. לדוגמה:
- תזרים: יציאה $430,000, קטגוריה "שיפוץ", תת-קטגוריה "הריסה ובנייה"
- → מתעדכן אוטומטית: עלויות שיפוץ > הריסה ובנייה > בפועל = $430,000

##### שדות מחושבים אוטומטית
| שדה | חישוב |
|------|--------|
| סך השקעה מתוכנן | סכום כל הקטגוריות (תכנון) |
| סך השקעה בפועל | סכום כל הקטגוריות (בפועל) |
| חריגה כוללת | בפועל − תכנון |
| % חריגה כולל | (חריגה / תכנון) × 100 |
| רווח נקי צפוי (תכנון) | מחיר מכירה צפוי − סך השקעה מתוכנן |
| רווח נקי צפוי (מעודכן) | מחיר מכירה צפוי − סך השקעה בפועל |
| אחוז תשואה (תכנון) | (רווח תכנון / סך תכנון) × 100 |
| אחוז תשואה (מעודכן) | (רווח מעודכן / סך בפועל) × 100 |

#### טאב ד׳: תזרים מזומנים (Cash Flow)

טבלה דינמית שבה המנהל מוסיף שורות של תנועות כספיות. **התזרים מחובר למחשבון הפיננסי** — כל הוצאה עם קטגוריה מעדכנת אוטומטית את הסכום "בפועל" בטאב ב׳.

| שדה | סוג | הערות |
|------|------|-------|
| תאריך | תאריך | **חובה** — תאריך התנועה בפועל |
| סוג | בחירה | כניסה / יציאה |
| סכום | מספר ($) | |
| מהות/תיאור | טקסט | למשל: "תשלום לקבלן Phase 3", "קבלת השקעה מאבי" |
| קטגוריה | בחירה | מתוך קטגוריות העלות שהוגדרו בטאב ב׳ (רכישה / שיפוץ / מימון / החזקה / מכירה / אחר) |
| תת-קטגוריה (פריט) | בחירה | מתוך פריטי העלות בקטגוריה שנבחרה (לדוגמה: אם קטגוריה = "שיפוץ" → "הריסה ובנייה" / "חומרי גמר" / "פיתוח חוץ") |
| תאריך הזנה | אוטומטי | תאריך ושעה שבהם השורה הוזנה למערכת (audit) |

> **כלל חשוב**: כל פעולה פיננסית חייבת לכלול תאריך. אין אפשרות לשמור תנועה ללא תאריך.

> **חיבור אוטומטי**: כשהמשתמש מזין הוצאה עם קטגוריה + תת-קטגוריה, הסכום "בפועל" של אותו פריט בטאב ב׳ מתעדכן אוטומטית. המערכת צוברת: אם יש כמה תנועות לאותו פריט, הסכום בפועל = סכום כל התנועות.

> **התראות**: [לבירור — האם בכל פעם שמוזנת הוצאה או הכנסה, המערכת צריכה לשלוח התראה? למי? באיזה ערוץ? (אימייל/וואטסאפ/push)]

##### שדות מחושבים אוטומטית
| שדה | חישוב |
|------|--------|
| סה"כ כניסות | סכום כל שורות מסוג "כניסה" |
| סה"כ יציאות | סכום כל שורות מסוג "יציאה" |

## שלב 2: דאשבורד משקיעים (Investor Portal) — `/investor`

ממשק נפרד למשקיעים. המשקיע מתחבר עם אימייל + סיסמה ורואה **רק** את העסקאות שהוא משקיע בהן.

### מסך ראשי — העסקאות שלי

רשימת כרטיסים של העסקאות שהמשקיע משתתף בהן:
- שם הנכס + תמונה
- סטטוס נכס (איתור/שיפוץ/מכירה...)
- סכום ההשקעה שלי
- אחוז גיוס כולל

### מסך עסקה — תצוגת משקיע

לחיצה על עסקה מציגה:

#### סקשן 1: פרטי הנכס
- שם, כתובת, תמונה ראשית
- סטטוס נכס + timeline (כמו באתר הציבורי)
- תיאור העסקה

#### סקשן 2: התקדמות פיננסית
- **תזרים מזומנים** — טבלת הכנסות/הוצאות (read-only) עם מאזן
- **סיכום**: סה"כ כניסות, סה"כ יציאות, מאזן נוכחי
- **גרף תזרים** (אופציונלי) — גרף עמודות של כניסות/יציאות לאורך זמן

#### סקשן 3: עדכונים מהשטח
- גלריית תמונות וסרטונים שהאדמין העלה (מטאב י׳)
- מסודר לפי תאריך, חדש למעלה
- כל פריט עם כותרת + תאריך

#### סקשן 4: מסמכים
- רשימת מסמכים שהאדמין סימן כ"גלוי למשקיעים" (חוזה LLC, הסכם רכישה וכו׳)
- אפשרות להוריד כל מסמך

#### סקשן 5: פרטי ההשקעה שלי
- סכום ההשקעה שלי
- תאריך ההשקעה
- אחוז בעלות (אם מוגדר)

> **חשוב**: המשקיע רואה הכל ב-read-only. הוא לא יכול לערוך שום דבר. רק האדמין מנהל את הנתונים.

## אבטחה

- **מערכת roles עם 3 רמות**:
  - **super_admin** (יחיד) — גישה מלאה + ניהול משתמשים והרשאות
  - **manager** (שותפים) — ניהול עסקאות, תזרים, מסמכים. ללא גישה לניהול הרשאות
  - **investor** [שלב 2] — read-only רק לעסקאות שמשויכות אליו
- הגנת סיסמה עם JWT (session token)
- אין גישה ללא אימות — כל route מוגן
- Session timeout אחרי חוסר פעילות
- סיסמאות נשמרות כ-hash (bcrypt)
- **Audit log** — כל פעולה פיננסית (הוצאה/הכנסה/עדכון) מתועדת עם: מי ביצע, מתי, מה השתנה

## שדות מחושבים (לא נשמרים ב-DB — מחושבים בזמן אמת)

### מחשבון פיננסי — תכנון
| שדה | חישוב |
|------|--------|
| סכום קטגוריה (תכנון) | SUM של planned_amount בכל cost_items של הקטגוריה |
| סך השקעה מתוכנן | SUM של כל סכומי הקטגוריות (תכנון) |
| רווח נקי צפוי (תכנון) | expected_sale_price − סך השקעה מתוכנן |
| אחוז תשואה (תכנון) | (רווח תכנון / סך תכנון) × 100 |

### מחשבון פיננסי — בפועל
| שדה | חישוב |
|------|--------|
| סכום קטגוריה (בפועל) | SUM של actual_amount בכל cost_items של הקטגוריה |
| סך השקעה בפועל | SUM של כל סכומי הקטגוריות (בפועל) |
| רווח נקי מעודכן | expected_sale_price − סך השקעה בפועל |
| אחוז תשואה מעודכן | (רווח מעודכן / סך בפועל) × 100 |

### השוואה תכנון מול ביצוע
| שדה | חישוב |
|------|--------|
| חריגה per פריט | actual_amount − planned_amount |
| % חריגה per פריט | (חריגה / planned) × 100 |
| חריגה per קטגוריה | SUM חריגות הפריטים |
| חריגה כוללת | סך בפועל − סך תכנון |
| סטטוס חריגה | ירוק (≤0%), צהוב (1-10%), אדום (>10%) |

### גיוס הון
| שדה | חישוב |
|------|--------|
| הון שגויס | SUM של deal_investors.amount |
| אחוז גיוס | (הון שגויס / fundraising_goal) × 100 |
| הון שנותר | fundraising_goal − הון שגויס |

### תזרים מזומנים
| שדה | חישוב |
|------|--------|
| סה"כ כניסות | SUM של cashflow WHERE type = 'income' |
| סה"כ יציאות | SUM של cashflow WHERE type = 'expense' |
| מאזן נוכחי | כניסות − יציאות |
| סיכום per קטגוריה | SUM של cashflow GROUP BY category_id |

### Comps — השוואה
| שדה | חישוב |
|------|--------|
| מחיר לרגל (comp) | sale_price / sqft |
| מרחק מהנכס | Haversine formula על lat/lng |
| הפרש מחיר | comp sale_price − our ARV |
| הפרש $/sqft | comp $/sqft − our $/sqft |
| ציון דמיון | weighted score: מרחק (40%) + גודל (25%) + חדרים (20%) + שנת בנייה (15%) |

## הערות למימוש

1. **האתר הציבורי** ימשיך לעבוד כ-static HTML — הדאשבורד ישמור את הנתונים ויייצר/יעדכן את ה-HTML של האתר (או שהאתר יקרא מ-API)
2. **קישור Zillow** — בשלב ראשון רק שדה URL. בעתיד אפשר להוסיף scraping אוטומטי של פרטים ותמונות
3. **תמונות** — בשלב ראשון URL חיצוני. בעתיד אפשר להוסיף העלאה ישירה לשרת
4. **גיבוי** — ה-DB הוא קובץ SQLite יחיד, קל לגבות
5. **תצוגה מקדימה** — כפתור שמייצר את ה-HTML של העסקה ומראה אותו ב-iframe
6. **תאריכים** — כל פעולה פיננסית חייבת תאריך. המערכת שומרת גם timestamp של ההזנה (audit)
7. **הפרדת הרשאות** — middleware שבודק role בכל route: super_admin רואה הכל, manager מנהל עסקאות, investor רואה רק את שלו
8. **חילוץ נתונים** — המערכת תומכת בקריאת XLSX ישירה (xlsx parser), חילוץ טקסט מ-PDF (pdf-parse), ו-OCR לסריקות (Tesseract.js או API חיצוני). ה-AI parsing משתמש ב-Claude API לזיהוי שדות מטקסט לא מובנה
9. **מיפוי מחשבון** — המבנה של קובץ ה-XLSX הקיים (מחשבון.xlsx) ממופה מראש: המערכת יודעת באילו תאים נמצא כל שדה (מחיר רכישה, ARV, עלויות שיפוץ וכו׳). עבור מחשבונים עם מבנה שונה — AI parsing
